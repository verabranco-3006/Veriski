# Event Management & Alert Validation — Working Plan

*This is the working document for strategy, design, and implementation notes.*

---

## Current State

**Problem Identified:**
Currently in RDINC, we collect both customer-reported incidents (support case escalations) and alerts (system-wide impact alerts, service-specific SLO alerts). Both receive the same incident response treatment, but alerts can have false positives.

**Current Model:**
- Alerts and incidents treated identically in RDINC
- No validation step before triggering incident response for alerts
- False positives consume incident response capacity
- No feedback loop to improve alert quality

**Impact:**
- Incident response noise reduces effectiveness
- False positives delay response to real incidents
- Alert quality issues persist without visibility
- Unclear separation between alert validation and incident declaration

---

## Target State

**New Model: Formal Event Management Practice**

### Alert Types in Scope

**1. System-Wide Impact Alerts**
- Alerts indicating potential platform-wide or multi-customer impact
- Typically generated by SRE monitoring systems
- High urgency when valid, but can have false positives

**2. Service-Specific SLO Alerts**
- SLO violations for specific services monitored by Development teams
- Can range from actual customer impact to monitoring configuration issues
- Require validation before escalating to incident response

**3. Development Team Alerts**
- Custom alerts created by individual development teams
- May have varying quality and tuning
- Need consistent validation framework

### Event Management Practice Components

**1. Alert Intake Process**
- Separate intake channel from customer-reported incidents
- Integration points with monitoring tools (PagerDuty, Grafana, etc.)
- Initial categorization by alert type

**2. Validation Criteria (To Be Defined)**
- Framework for determining if alert requires incident response
- Criteria for identifying false positives
- Escalation thresholds based on severity/impact

**3. Validation Process**
- Who reviews alerts? (SRE on-call? Dedicated event management team? Automated with human oversight?)
- Review workflow and decision points
- Handoff to incident response when validated

**4. False Positive Handling**
- Documentation of false positive patterns
- Feedback loop to alert owners
- Alert tuning recommendations
- Alert deprecation process for persistent false positives

**5. RDINC Integration**
- Separate alert intake channel vs customer-reported incidents
- Different issue types or workflow states
- Enriched context for validated alerts entering incident response

## Validation Framework (Draft)

### Validation Decision Tree (To Be Refined)

**Step 1: Alert Classification**
- System-wide impact? → High priority validation
- Service-specific SLO? → Medium priority validation
- Development team custom alert? → Context-dependent validation

**Step 2: Impact Assessment**
- Is customer impact confirmed or highly likely?
  - YES → Trigger incident response immediately
  - NO → Proceed to Step 3

**Step 3: Historical Pattern Analysis**
- Has this alert triggered incidents before?
- What was the false positive rate historically?
- Are there known monitoring configuration issues?

**Step 4: Correlation Check**
- Are there other correlated alerts?
- Multiple systems affected?
- Sustained vs transient issue?

**Step 5: Validation Decision**
- **Trigger Incident Response:** High confidence of customer impact
- **False Positive:** No evidence of customer impact, known false positive pattern
- **Needs Investigation:** Unclear, requires deeper analysis before decision

## Measurement Framework

### M3.4: Alert Validation Accuracy (FORMAL V2MOM METRIC)
**Metric:** % of alerts where validation decision matches actual outcome

**Definition:** Retrospective accuracy measurement - did "Trigger Incident" decisions result in real incidents? Did "False Positive" decisions correctly filter non-incidents?

**Target:** >85% by Q2 2027

**Collection Method:** Quarterly retrospective analysis comparing validation decisions to actual outcomes

### M3.4a: Event Management Coverage (FOUNDATIONAL)
**Metric:** % of alerts processed through Event Management practice

**Definition:** Percentage of total alerts that go through validation vs. alerts that still go directly to incident response

**Target:** 100% by Q4 2026

**Collection Method:** Track alert source - count alerts through Event Management vs direct to RDINC

### M3.4b: Alert Volume & Classification Baseline (FOUNDATIONAL)
**Metric:** Monthly alert volume by source and outcome

**Definition:** Track total alerts received, categorized by:
- Source: System-wide impact / Service-specific SLO / Development team alerts
- Outcome: Triggered incident / False positive / Auto-resolved

**Target:** Baseline established by Q3 2026, ongoing monitoring

**Collection Method:** Monthly reporting from Event Management intake data

### M3.4c: False Positive Identification Rate (FOUNDATIONAL)
**Metric:** % of alerts identified as false positives by Event Management review

**Definition:** Percentage of alerts reviewed that are correctly identified as false positives and filtered before triggering incident response

**Target:** Baseline Q3 2026, >70% identification rate by Q4 2026

**Collection Method:** Track validation decisions - count "False Positive" decisions as % of total alerts reviewed

### M3.4d: Event Review Lead Time (FOUNDATIONAL)
**Metric:** Mean time from alert received to Event Management validation decision

**Definition:** Average time from alert ingestion to validation decision (Trigger Incident / False Positive / Needs Investigation)

**Target:** <15 minutes by Q4 2026

**Collection Method:** Timestamp tracking - alert received time to validation decision time

## Integration with RDINC

### Current RDINC Workflow
1. Artifact created (incident or alert)
2. Triage (classification, severity, assignment)
3. Investigation & resolution
4. Closure

### Proposed Event Management Integration

**Option 1: Separate Issue Type**
- Create "Alert" issue type in RDINC separate from "Incident"
- Alert workflow: Intake → Validation → [Trigger Incident / Close as False Positive]
- If "Trigger Incident", convert to Incident issue or create linked Incident

**Option 2: Pre-RDINC Validation**
- Alerts validated before RDINC entry
- Only validated alerts create RDINC incidents
- False positives logged separately (not in RDINC)

**Option 3: Validation Workflow State**
- Alerts enter RDINC in "Validation Pending" state
- Event Management moves to "Validated - Proceed" or "False Positive - Close"
- Standard incident response only for "Validated - Proceed"

**Recommendation (To Be Decided):** TBD based on RDINC team input and pilot learnings

## False Positive Feedback Loop

### Feedback Mechanism (Draft)

**1. Pattern Documentation**
- Document each false positive with:
  - Alert details (source, type, threshold)
  - Why it was false positive (monitoring issue, threshold too low, etc.)
  - Recommended tuning action

**2. Alert Owner Notification**
- Weekly/monthly summary report to alert owners
- Include:
  - False positive count and rate
  - Pattern analysis
  - Tuning recommendations
  - Impact on incident response efficiency

**3. Tuning Tracking**
- Track which recommendations are implemented
- Measure reduction in false positives after tuning
- Close the loop with alert owners on effectiveness

**4. Alert Quality Score (Potential)**
- Score alerts based on false positive rate
- Prioritize tuning for lowest quality alerts
- Consider deprecating persistently low-quality alerts

## Risks & Mitigation Strategies

**Risk 1: Validation delays real incidents**
- **Impact:** High - could slow incident response
- **Mitigation:** Target <15 min review time (M3.4d), escalation path for high-severity alerts, parallel processing during pilot

**Risk 2: Validation ownership unclear**
- **Impact:** Medium - may cause adoption delays
- **Mitigation:** Define ownership model in Q2 2026 practice definition phase, align with SRE leadership early

**Risk 3: Integration complexity with RDINC**
- **Impact:** Medium - could slow pilot and scale phases
- **Mitigation:** Pilot approach (Q3 2026) validates integration before full scale, involve RDINC team in design

**Risk 4: Alert owners resist feedback loop**
- **Impact:** Medium - limits long-term quality improvement
- **Mitigation:** Frame as service quality improvement, provide actionable tuning recommendations, track improvement over time

**Risk 5: Insufficient validation capacity**
- **Impact:** Medium - could create validation backlog
- **Mitigation:** Scale validation ownership model based on alert volume, automate where possible, prioritize by alert severity

**Risk 6: False positive criteria too aggressive**
- **Impact:** High - could miss real incidents
- **Mitigation:** Conservative criteria initially, refine based on accuracy data (M3.4), continuous improvement based on feedback
