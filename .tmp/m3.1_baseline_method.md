# M3.1 Baseline Establishment Method

**Metric:** Public MTTA (Mean Time to post on external status page after system-wide declaration)

**Target:** < 30 minutes by Q4 2026

**Baseline Period:** Q4 2024 + Q1 2025 (post IC Leader rollout in Sept 2024)

---

## Data Sources Available

### RDINC Project (System-wide incidents)
- **RootlyID / RootlyKey** — Link to Rootly incident
- **Incident Type** = "System-wide"
- **Created Date, Resolved Date**

### MIM Project (Service Controllers - Major Incidents)
- **Status page published at** — Timestamp when external status page was posted
- **Linked RDINC ticket** (if available)

### Rootly (SRE)
- **System-wide declaration timestamp** — Pending confirmation from Kevin if available via API

---

## Baseline Method

### Step 1: Extract RDINC System-wide Incidents

**JQL Query:**
```jql
project = RDINC
AND "Incident Type" = "System-wide"
AND resolved >= "2024-10-01" AND resolved <= "2025-03-31"
ORDER BY resolved DESC
```

**Extract fields:**
- RDINC Key (e.g., RDINC-12345)
- RootlyID
- Created Date
- Resolved Date

**Export to CSV or Excel**

---

### Step 2: Identify MIM Tickets with Status Page Data

**Option A: If RDINC-MIM link exists**

**JQL Query:**
```jql
project = MIM
AND "Status page published at" IS NOT EMPTY
AND created >= "2024-10-01" AND created <= "2025-03-31"
ORDER BY created DESC
```

Extract fields:
- MIM Key
- Status page published at (timestamp)
- Linked RDINC Key (if available)

**Option B: If no link exists — Manual cross-reference**

For each RDINC incident:
1. Check incident title/description for identifying info (date, service, impact)
2. Search MIM project for matching incident by date/title
3. Extract "Status page published at" timestamp from matching MIM ticket

---

### Step 3: Extract Declaration Timestamp

**Option A: Via Rootly API (if Kevin confirms available)**

For each RootlyID from RDINC:
1. Call Rootly API: GET /incidents/{RootlyID}
2. Extract system-wide declaration timestamp field
3. Record timestamp

**Option B: Manual extraction from Slack (fallback)**

For each incident:
1. Find incident thread in #product-leadership-incidents-alerts
2. Identify system-wide declaration message timestamp
3. Record timestamp manually

**Option C: Use Rootly incident creation time as proxy (least accurate)**

If neither A nor B available:
- Use RDINC "Created Date" as proxy for declaration time
- Note: This assumes minimal delay between Rootly creation and Jira sync
- Add caveat to baseline: "Declaration timestamp estimated via incident creation time"

---

### Step 4: Match Data and Calculate MTTA per Incident

Create spreadsheet with columns:

| RDINC Key | RootlyID | Declaration Timestamp | MIM Key | Status Page Published At | MTTA (minutes) | Notes |
|-----------|----------|----------------------|---------|--------------------------|----------------|-------|
| RDINC-123 | 44326 | 2024-11-15 10:00:00 | MIM-2031 | 2024-11-15 10:15:00 | 15 | Complete data |
| RDINC-124 | 44327 | 2024-11-20 14:30:00 | MIM-2032 | 2024-11-20 15:10:00 | 40 | Exceeded target |

**Calculate MTTA per incident:**
```
MTTA = (Status Page Published At - Declaration Timestamp) in minutes
```

**Filter:**
- Include only incidents where "Status page published at" is NOT NULL (exclude incidents not posted externally)
- Exclude outliers where MTTA > 120 min (investigate separately as anomalies)

---

### Step 5: Calculate Baseline Metrics

**Primary Metric:**
```
Baseline MTTA = Mean(MTTA for all incidents)
```

**Supporting Metrics:**
- **Median MTTA** (less sensitive to outliers)
- **Min/Max MTTA** (range of performance)
- **% incidents < 30 min** (target adherence rate)
- **Sample size** (number of incidents with complete data)

**Expected Outcome:**
- Baseline MTTA: 15-30 min (hypothesis)
- Sample size: 10-20 incidents (depends on system-wide incident frequency)

---

## Data Completeness Assessment

For baseline to be valid, assess:

**Completeness Rate:**
```
Completeness = (Incidents with complete data / Total system-wide incidents) * 100
```

**Target:** > 70% completeness for reliable baseline

**Gaps to document:**
- Missing MIM tickets (system-wide incident not posted externally?)
- Missing declaration timestamps (Rootly data unavailable?)
- Unmatched RDINC-MIM pairs (manual matching required)

---

## Validation Checks

### Check 1: Spot-check Sample
- Pick 3-5 incidents
- Manually verify timestamps against Slack threads
- Confirm MTTA calculation is accurate

### Check 2: Outlier Investigation
- Any incident > 60 min? Investigate why:
  - Complex multi-region incident?
  - IC Leader unavailable?
  - Non-standard communication criteria?

### Check 3: Data Quality
- Are declaration timestamps reasonable? (Not before incident creation, not after external post)
- Are external post timestamps reasonable? (After declaration, before incident resolution)
